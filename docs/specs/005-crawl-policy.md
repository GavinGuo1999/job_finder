# Spec 005：采集策略与合规边界

## 目的

定义公开岗位采集的允许范围、禁止行为、失败处理和测试边界，确保项目只采集公开招聘信息，不绕过网站保护机制。

## 范围

包含：

* 允许采集的来源。
* 禁止行为。
* 请求频率和失败处理。
* CI 与 fixture 策略。

不包含：

* 具体 HTTP 客户端实现。
* 反爬绕过策略。
* 登录态或个人账户自动化。

## 输入

来自：

```text
data/sources/company_sources.csv
```

关键字段：

* `source_url`
* `source_type`
* `adapter`
* `enabled`

## 输出

每个来源应产生以下状态之一：

| 状态 | 说明 |
| --- | --- |
| `ok` | 成功读取并解析至少一个岗位。 |
| `empty` | 成功读取，但没有岗位。 |
| `failed` | 请求、解析或数据校验失败。 |
| `manual_review` | 需要人工检查，不自动继续。 |
| `skipped` | 来源被禁用或策略禁止访问。 |

最终写入 CSV 的相关字段：

* `source__status`
* `source__fetched_at`
* `debug__adapter`
* `debug__missing_fields`
* `debug__notes`

## 允许行为

* 访问公开、无需登录的 ATS API 或岗位页面。
* 访问公司公开招聘页。
* 使用合理低频请求。
* 尊重 `robots.txt` 和来源页面限制。
* 使用本地 fixture 测试解析器。

## 禁止行为

* 绕过验证码。
* 绕过登录墙。
* 使用代理池规避封禁。
* 使用盗取或个人账户 Cookie、Token。
* 高频请求压垮网站。
* 自动投递岗位。
* 采集候选人个人隐私数据。
* 访问明确禁止自动访问的页面。

## 失败处理

* 单个来源失败不得导致整次运行失败。
* 失败来源应记录状态和简短原因。
* 遇到登录、验证码、强反爬、403 或需要人机验证时，标记为 `manual_review` 或 `skipped`。
* 字段缺失时尽量保留岗位记录，并在 `debug__missing_fields` 标记。

## 验收标准

* 文档和后续实现不包含任何反爬绕过方案。
* CI 不访问真实网站。
* 来源失败时仍能输出 `jobs.csv`。
* 空岗位列表不报错。
* 人工检查状态可以在输出中看到。

## 测试要求

后续实现需覆盖：

* fixture 模拟成功、空列表、HTTP 失败、解析失败。
* 需要登录或验证码的 fixture 被标记为 `manual_review`。
* 单个来源失败不影响其他来源。
* 所有来源失败时仍输出只有表头的 CSV。
