# ADR 0001：第一阶段采用 CSV-first 架构

## 状态

已接受

## 日期

2026-06-29

## 背景

本项目的目标是构建一个本地自用的岗位机会雷达系统，用于采集公开岗位、标准化字段、打分排序，并帮助用户筛选值得关注和投递的岗位。

项目后续可能会接入 PostgreSQL、Metabase、分析报告和 BI Dashboard。

但在第一阶段，用户更关心：

* 快速看到结果
* 能用 Excel / WPS 打开检查
* 能按分数排序
* 能人工判断评分是否合理
* 能让 Codex/Copilot 容易验收
* 避免过早引入数据库和 BI 复杂度

因此需要决定：第一阶段是否直接使用数据库，还是先使用 CSV 输出。

## 可选方案

### 方案 A：直接使用 PostgreSQL

优点：

* 结构化程度高
* 适合历史追踪
* 适合后续接 Metabase
* 适合复杂查询

缺点：

* 初始工程复杂度更高
* 需要 migration、连接配置、导入导出逻辑
* 不利于第一阶段快速人工验收
* 容易让项目过早偏向数据库工程，而不是岗位数据质量

### 方案 B：第一阶段使用多个 CSV 文件

优点：

* 分层清晰
* 接近数据库表结构
* 方便后续导入数据库

缺点：

* 用户每次验收需要打开多个文件
* 对 MVP 来说不够直观
* 容易在早期增加理解成本

### 方案 C：第一阶段只输出一个岗位大宽表 CSV

优点：

* 一个文件即可验收
* 一行一个岗位，直观清晰
* 可以直接排序、筛选、打分
* 方便用户人工检查 top 岗位
* 后续仍可导入数据库或 Metabase
* 降低第一阶段复杂度

缺点：

* 字段较多，CSV 会变宽
* 技能、风险、来源等信息会被扁平化
* 不如关系型数据库适合长期历史追踪
* 后续可能需要拆表

## 决策

第一阶段采用方案 C：

```text
CSV-first，一次运行只输出一个岗位大宽表。
```

输出路径：

```text
outputs/runs/<timestamp>/jobs.csv
```

每一行代表一个岗位。

字段使用前缀分组：

```text
run__
source__
company__
job__
location__
language__
comp__
benefit__
text__
skill__
risk__
score__
debug__
```

## 后果

正面影响：

* 第一阶段更容易完成
* 更容易给 Codex/Copilot 验收
* 用户可以快速打开 CSV 检查结果
* 更容易调整评分规则
* 更容易迭代岗位字段

负面影响：

* CSV 会比较宽
* 长文本字段可能影响 Excel 打开体验
* 技能、来源、风险信息暂时不做关系型拆表
* 后续接数据库时需要设计导入映射

## 后续演进

当 CSV-first MVP 稳定后，再考虑：

1. 导入 PostgreSQL
2. 拆分 jobs、companies、sources、skills、scores 等表
3. 接入 Metabase
4. 增加历史趋势分析
5. 增加日报/周报
