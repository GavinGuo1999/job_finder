# ADR 0002：评分采用配置驱动

## 状态

已接受

## 日期

2026-06-29

## 背景

Job Radar 的排序结果需要反复人工校准。用户会根据真实 `jobs.csv` 输出检查前 50 个岗位是否符合直觉，并调整地区、公司、岗位、语言、技能和风险的权重。

如果评分规则硬编码在实现里，每次调整都需要改代码和重新测试，不利于快速调权。

## 可选方案

### 方案 A：硬编码评分规则

优点：

* 实现简单。
* 类型和分支更容易直接控制。

缺点：

* 调整成本高。
* 用户难以直接审查。
* 容易把个人偏好散落在代码中。

### 方案 B：使用数据库配置评分

优点：

* 后续可做管理界面。
* 适合多人协作和历史版本管理。

缺点：

* 第一阶段引入数据库过重。
* 不符合 CSV-first 的 MVP 目标。

### 方案 C：使用 `data/config/*.csv` 配置评分

优点：

* 可人工编辑。
* 与 CSV-first 架构一致。
* 容易测试和版本化。
* 不需要数据库或 Web UI。

缺点：

* 配置校验需要额外测试。
* 复杂规则不如代码灵活。

## 决策

采用方案 C：第一阶段评分规则由 `data/config/*.csv` 驱动。

具体评分字段、默认分值和总分公式记录在：

```text
docs/specs/008-scoring.md
```

## 后果

正面影响：

* 用户可以直接调整评分。
* 评分更透明，可解释性更强。
* 后续可以把 CSV 配置迁移到数据库。

负面影响：

* 需要严格校验配置表。
* 需要处理未知枚举和空值。
* 过复杂的条件规则需要谨慎设计，避免配置表变成隐式代码。
